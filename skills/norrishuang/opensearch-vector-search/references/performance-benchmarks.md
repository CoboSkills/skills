# OpenSearch 向量搜索性能基准

## 核心概念

本文档提供 OpenSearch 向量搜索在不同实例类型、数据集规模和向量维度下的性能基准数据，帮助选择合适的配置方案。包含详细的成本对比分析，帮助在性能和成本之间做出最优决策。

## 关键性能指标说明

- **QPS (Queries Per Second)**: 每秒查询数，越高越好
- **Latency (p95/p99)**: 延迟百分位数，越低越好
- **Recall@10/100**: 召回率，越高越好（通常 > 0.9 为优秀）
- **ef_search**: HNSW 搜索参数，影响召回率和性能平衡
- **Load Time**: 索引构建时间（秒）

## 量化压缩 (Binary Quantization)

### BQ 性能对比 (1M 向量, 1 × r8g.4xlarge)

#### 标准模式 (无压缩)

| Shards | ef_search | QPS | Latency (p95) | Recall@100 |
|--------|-----------|-----|---------------|------------|
| 1 | 100 | 4889.04 | 6.5ms | 0.9697 |
| 1 | 200 | 4294.15 | 6.1ms | 0.9812 |

#### BQ 1-bit (oversample-factor 20)

| Shards | ef_search | QPS | Latency (p95) | Recall@100 | 性能变化 |
|--------|-----------|-----|---------------|------------|----------|
| 1 | 100 | 3097.73 | 7.9ms | 0.7659 | -37% QPS, -21% Recall |
| 8 | 100 | 570.04 | 11.4ms | 0.9243 | 多分片补偿 |

#### BQ 4-bit (oversample-factor 20)

| Shards | ef_search | QPS | Latency (p95) | Recall@100 | 性能变化 |
|--------|-----------|-----|---------------|------------|----------|
| 8 | 100 | 541.84 | 11.8ms | 0.9598 | 高召回率 |
| 8 | 200 | 461.32 | 13.1ms | 0.9838 | 接近无损 |

**关键发现**:
- BQ 1-bit: 内存节省 32×，但召回率下降明显
- BQ 4-bit: 内存节省 8×，召回率接近无损
- oversample-factor 从 20 降到 10 可提升 QPS 约 54%
- 多分片配置可补偿量化带来的性能损失

### 按性能需求选择（含成本权衡）

**低延迟场景** (< 10ms):
- 使用内存模式
- ef_search = 40-100
- 选择高频 CPU 实例 (r7g/r8g)
- 避免磁盘模式
- **成本**: 高（基准）
- **适用**: 实时推荐、搜索服务

**高吞吐场景** (> 2000 QPS):
- 增加 shards 数量
- 使用大实例 (4xlarge+)
- 考虑多副本分散负载
- **成本**: 很高（基准 × 2-3）
- **适用**: 高流量应用

**成本优化场景**:
- 使用磁盘模式 + 量化压缩
- 接受更高延迟 (100-200ms)
- 适合批处理或离线场景
- **成本**: 低（节省 50-80%）
- **适用**: 批处理、数据分析、MVP 阶段

**高召回场景** (Recall > 0.95):
- ef_search = 200-400
- ef_construction = 400+
- 避免过度量化压缩
- **成本**: 中高（基准 × 1.2-1.5）
- **适用**: 精准匹配、关键业务

**平衡场景**（推荐）:
- 使用 8x 量化压缩
- ef_search = 100-200
- 内存模式 + 合理分片
- **成本**: 中（节省 30-50%）
- **适用**: 大多数生产环境

## 参数调优建议

### HNSW 参数

**ef_construction**:
- 默认: 128
- 高质量索引: 200-400
- 权衡: 更高值增加索引时间但提升召回率

**m (连接数)**:
- 默认: 16
- 高维数据: 20-24
- 权衡: 更高值增加内存但提升召回率

**ef_search**:
- 低延迟: 40-100
- 平衡: 100-200
- 高召回: 200-400

### 量化压缩参数

**oversample-factor**:
- BQ 1-bit: 10-20
- BQ 4-bit: 20
- 权衡: 更高值提升召回但降低性能

**压缩比选择**:
- 32x (1-bit): 极端内存受限，可接受召回损失
- 8x (4-bit): 推荐，召回率接近无损
- 无压缩: 内存充足，追求最高性能

### 分片策略

**Shards 数量**:
- 小数据集 (< 10M): 2-4 shards
- 中等数据集 (10M-50M): 6-9 shards
- 大数据集 (> 50M): 10-17 shards
- 经验公式: shards ≈ CPU 核心数 × 1.5

**Replica 数量**:
- 开发环境: 0 replica
- 生产环境: 1-2 replica
- 高可用: 2 replica (3 AZ)

## 常见问题

### 问题 1: QPS 不稳定，波动很大

**症状**: QPS 从几十到上千波动，延迟也大幅波动

**原因**:
- 磁盘模式下 EBS IOPS 限制
- 内存不足导致频繁换页
- JVM GC 压力过大

**解决方案**:
1. 检查 EBS IOPS 和吞吐量配置
2. 增加 IOPS 到 9000+，吞吐量到 500MB/s+
3. 监控内存使用，考虑升级实例
4. 调整 JVM heap 大小

### 问题 2: 召回率低于预期

**症状**: Recall@100 < 0.9

**原因**:
- ef_search 设置过低
- 量化压缩损失过大
- 索引质量不足

**解决方案**:
1. 提高 ef_search 到 200+
2. 使用 4-bit 替代 1-bit 量化
3. 提高 ef_construction 到 400+
4. 增加 oversample-factor

### 问题 3: 内存占用过高

**症状**: 内存使用 > 85%，频繁 OOM

**原因**:
- 数据集过大
- 未使用压缩
- 副本过多

**解决方案**:
1. 启用量化压缩 (8x 或 32x)
2. 使用磁盘模式
3. 减少副本数量
4. 升级到内存优化实例

### 问题 4: 索引构建时间过长

**症状**: Load Time > 10 小时

**原因**:
- 数据集过大
- ef_construction 设置过高
- 单线程写入

**解决方案**:
1. 启用 GPU 加速（提升 45-81%）
2. 降低 ef_construction 到 128-200
3. 增加写入线程数
4. 使用更多 shards 并行构建

### 总体成本优化策略

#### 成本优化优先级

1. **使用 8x 量化压缩** → 节省 50-75%
   - 召回率几乎无损
   - 立即生效
   - 无需架构变更

2. **优化副本配置** → 节省 33-50%
   - 开发环境使用 0 副本
   - 生产环境使用 1 副本
   - 避免过度冗余

3. **使用 Reserved Instances** → 节省 30-50%
   - 1 年预留适合大多数场景
   - 3 年预留适合核心服务
   - 需要长期承诺

4. **考虑磁盘模式** → 节省 50-80%
   - 仅适合批处理场景
   - 需要接受高延迟
   - 需要高 IOPS EBS

5. **右调实例大小** → 节省 20-40%
   - 监控资源使用率
   - CPU/内存目标 60-80%
   - 定期评估和调整

#### 成本优化决策树

```
是否可接受 5% 召回率损失？
├─ 是 → 使用 8x 压缩（推荐）
│   └─ 是否为生产环境？
│       ├─ 是 → 1 副本 + 1 年 RI → 总节省 65%
│       └─ 否 → 0 副本 → 总节省 75%
└─ 否 → 是否可接受 100ms+ 延迟？
    ├─ 是 → 磁盘模式 + 32x 压缩 → 总节省 80%
    └─ 否 → 标准配置 + 1 年 RI → 总节省 30%
```
